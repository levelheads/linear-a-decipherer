## Executive Summary

The KH `*86` object-class split is not an isolated anomaly. It sits inside a broader KH support ecology in which nodule and roundel vocabularies are sharply separated.

Using the local corpus objects:

1. KH nodules (`KHWa`) have a small sole-content vocabulary set.
2. KH roundels (`KHWc`) have a much larger sole-content vocabulary set.
3. The two sets show **no overlap at all**, aside from one damaged/uncertain sign.

This materially strengthens the earlier `*86-RO` vs `*86+*188` result:

**The KH administration appears to segregate control labels by support class as a system property, not just in the `*86` family.**

## Findings

### 1. The `*86` split matches the broader KH support regime

The earlier result was:

- `*86-RO` on nodules
- `*86+*188` on roundels

This memo shows that this is not a quirky one-family accident. KH support classes are already strongly separated at the level of sole-content labels.

### 2. Nodule and roundel labels behave like different administrative repertoires

The KH nodules use a smaller and more repetitive label set. The roundels use a wider and more differentiated label set.

That is exactly what we would expect if the two supports served different administrative roles inside one system.

### 3. The `*86` family is a strong test case because both branches preserve the same sign base

Most support-separated labels do not share an obvious base sign. The `*86` family does.

That makes it unusually valuable:

- the system-level split is already visible across KH supports
- the `*86` family then gives a rare internal bridge across those supports

This is one reason the `*86` dossier is higher-value than generic single-sign cataloguing.

### 4. This result is methodological, not lexical

Nothing here solves `*86` phonetically or semantically. What it does is improve the document ontology:

- support matters
- support-specific labels matter
- the `*86` family should be interpreted inside that object ecology

This is a legitimate decipherment advance because it reduces false lexical branching later.

## Determination

The KH `*86` object-class split is now best understood as one case inside a **broader KH support ecology of segregated control labels**.

That means future KH work should default to:

- support-aware comparison first
- lexical speculation second

---

## Operational Consequences

1. Keep KH nodule and KH roundel control families in one comparative dossier.
2. Treat support class as a primary feature when evaluating new KH labels.
3. Do not merge KH nodule and roundel labels into one lexical bucket without direct evidence.
4. Use this ecology check as a guard against over-reading isolated KH signs.
