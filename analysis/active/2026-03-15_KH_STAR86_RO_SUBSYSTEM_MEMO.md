## Findings

### 1. `*86-RO` is a KH administrative family, not an isolated inscription problem

The four `KHWa` attestations are too formulaic and too concentrated to justify tablet-by-tablet translation work. The correct unit of analysis is the family:

- one sign base: `*86`
- one repeated ending: `-RO`
- one site: KH
- one narrow document class: `KHWa`

This is exactly the pattern the blocker lane should absorb.

### 2. The `-RO` ending is structurally interesting but still underdetermined

`-RO` invites comparison with known completion/closure behavior such as `KU-RO`, but the comparison is still only structural.

Current evidence does **not** support:

- equating `*86-RO` with `KU-RO`
- treating `RO` as a solved morpheme in this context
- promoting a lexical translation for the whole form

What it does support is a narrower statement: `*86-RO` likely belongs to the KH administrative control layer rather than the open lexicon.

### 3. `*86-RO` and `*86+*188` are best treated as parallel branches of the same subsystem

The working model that best fits the evidence is:

- `*86+*188` = KH roundel/receipt stamp family
- `*86-RO` = KH tablet-side control or closure family

This remains a structural model, not a decipherment claim. The important point is that both forms are administrative and KH-specific, which lowers the value of spending reading throughput on them as if they were ordinary commodity lines.

### 4. No translation breakthrough is justified here

There is still no evidence that `*86-RO` carries:

- a recoverable spoken lexical item
- a commodity identity
- a person or place name

The best current use of the family is as a constraint on later KH readings: when `*86` appears in KH administrative contexts, the default should be subsystem/control behavior first, not open lexical interpretation.
